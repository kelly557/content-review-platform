Unified Moderation API / Business Review Pack

统一审核 API 业务流程图与数据流转图

这份 HTML 将现有接口设计转成更适合评审的视觉文档：先给出整体树状视图，再展开同步审核、异步媒体审核、结果聚合和对象级数据流。重点不是“接口字段清单”，而是“业务怎么走、数据怎么拆、结果怎么合”。

设计方向
Swiss。用红色 folio、可见网格和强信息层级，让复杂链路一眼分区。
适用人群
产品、后端、风控、审核策略、AI 网关、媒体处理平台。
对应规范
moderation-openapi.yaml

目录

01

统一入口

对外只保留同步检查和异步任务两类入口，避免调用方按模态接多套协议。

02

媒体预处理

音视频不直接审核原始流，而是拆成 ASR、OCR、抽帧、文件元数据等可审核对象。

03

统一结果模型

`suggestion + details[] + aggregation` 是所有模态共享的最终语义中心。

最终结论不只给动作，还给 `winningDetails` 和 `notes`，支持审计与复盘。

01

总览树与范围

先把接口边界讲清楚：哪些走同步，哪些走异步，最终都收敛到同一套决策模型。

同步短文本、单图、图文、小文件，单次在线请求可完成。
异步音频、视频、大文件、批量、多阶段链路，需要任务编排和时间轴证据。
统一出口无论链路复杂度如何，最终都回到 suggestion、details、aggregation。

03

同步审核流程

同步链路的重点是低延迟和统一返回。它不处理原始音视频，而是专注在线实时可完成的审核。

入口控制认证、限流、schema 校验、scene 策略绑定。
能力路由按 input.type 选择文本、图片、图文、文件等审核引擎。
返回重点业务动作优先看 suggestion，规则系统优先看 label。

05

结果聚合规则

统一结果模型真正有价值的地方，在于最终不是一堆离散命中，而是一个可执行、可解释、可审计的动作结论。

Priority
`block` > `mask` > `watch` > `pass`
Winner
聚合器需要显式记录 `winningSuggestion` 与 `winningDetails`，不能只返回最终动作。
Level
最终 `level` 从胜出维度中选出最强值，而不是简单取平均或依赖单个 confidence。
Explainability
`notes` 用于解释为什么被 block / mask / watch，适合审计、申诉和策略复盘。

Aggregation Logic

先比较 detail.suggestion，再确定 winning details，最后输出可解释的 aggregation 对象。

flowchart TD
    A["details[]"] --> B["逐个读取 detail.suggestion"]
    B --> C{"是否存在 block?"}
    C -->|"是"| D["final suggestion = block"]
    C -->|"否"| E{"是否存在 mask?"}
    E -->|"是"| F["final suggestion = mask"]
    E -->|"否"| G{"是否存在 watch?"}
    G -->|"是"| H["final suggestion = watch"]
    G -->|"否"| I["final suggestion = pass"]
    D --> J["提取所有 winning details"]
    F --> J
    H --> J
    I --> J
    J --> K["计算 final level
从 winning details 取最强 level"]
    K --> L["生成 aggregation.policyId / policyVersion"]
    L --> M["输出 aggregation.winningSuggestion"]
    L --> N["输出 aggregation.winningDetails"]
    L --> O["输出 aggregation.notes"]
